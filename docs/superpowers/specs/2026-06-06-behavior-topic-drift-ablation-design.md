# Диагностика: дрифт темы на direct trigger — ablation-эксперимент

**Дата:** 2026-06-06
**Статус:** дизайн (диагностика, не фикс)
**Тип:** разовый эксперимент, local-only (не коммитим)

## Контекст / эпизод

В чате произошёл эпизод (DB id сообщений):

- `190`: «земля плоская»
- `191`: «вич несуществует»
- `192`: «кек»
- `193`: «накидал клоунов»
- `194-198`: ответы Даниила про землю/ВИЧ/клоунов
- `199`: «Карл объяснись» (direct trigger, прямое обращение к боту)
- `200`: ответ бота — **плохой**: вместо разбора свежей пачки (клоуны/земля/ВИЧ) бот уплыл в
  старую тему AI/джуны/агенты и склеил два фрейма («клоун/цирк» из свежего + «ИИ для рутины,
  люди для ревью» из старого).

Ответ выдан в `behavior_event 90` (реакция на `199`).

### Кандидатные причины (из ручного разбора)

1. **Нет явного topic-window для direct trigger** — последние сообщения видны, но не помечены
   как «текущая тема».
2. **Нет timestamps** в `messages`/prompt → паузы и смена темы невидимы.
3. «Карл объяснись» слишком короткое/анафоричное — модель сама выбирает, к чему «объяснись».
4. **Persistent state переусилил старый AI/джуны фрейм** (system-блоки personality/political).
5. Странная разметка `[to:room]` для прямого обращения (на момент ответа `selfIdentity.name`
   был `Bot`, не `Карл`, поэтому `addressedToSelf` не сработал).

### Подтверждённая механика причины #1 (из кода)

Для direct trigger контекст набирается через
[`drainForDirectTrigger`](../../../src/application/behavior/BehaviorGateBatcher.ts) →
возвращает только то, что **ещё лежит в батчере и не прогонялось через gate**
(`all.slice(-maxDirectContextMessages)`, затем `batches.delete`).
Сообщения `194-198` к моменту `199` уже были обработаны отдельными gate-событиями → вымело из
батчера → в `contextMessageIds` они не попали → `[GATE_CONTEXT]` для триггера пустой/тонкий.
При этом сами `194-198` **были в prompt** — ассемблер всегда тянет последние
`recentHistoryLimit` (80) сообщений
([`DefaultBehaviorContextAssembler.ts`](../../../src/application/behavior/DefaultBehaviorContextAssembler.ts)).

**Вывод:** контент присутствовал, не хватало **разметки** «это текущая тема». Это делает фикс
#1 разметочным (дешёвым), но прежде чем фиксить — эмпирически устанавливаем доминанту.

## Цель и критерий успеха

Установить, **правка какой причины убирает дрифт** ответа на `Карл объяснись` из свежей темы
(клоуны/земля/ВИЧ) в старую (AI/джуны/агенты).

**Gate воспроизводимости (критично):** сначала гоняем baseline `V0` (точный лог-prompt).
Если он **не дрейфит** в заметной доле (ориентир ≥30% из K=10) — эксперимент неинформативен
(исходный ответ был редким сэмплом), и ablation ничего не доказывает. Тогда либо растим K,
либо честно фиксируем «невоспроизводимо» и пересматриваем подход.

**Доминанта** = вариант с наибольшим падением drift-rate относительно `V0`.

## Метод: ablation на реальном лог-prompt

Берём точный prompt из `prompts.log`, делаем по варианту на причину (минимальная мутация),
гоняем каждый через ту же модель K раз, судим дрифт. Причина, чья правка чинит ответ, — доминанта.

### Получение входа

`prompts.log` пишется в [`DefaultBehaviorAiService.logPrompt`](../../../src/application/behavior/DefaultBehaviorAiService.ts)
в формате:

```
[<ISO timestamp>] behaviorDecision
PROMPT:
<JSON.stringify(messages, null, 2)>   // массив { role, content }
RESPONSE:
<JSON ответа>
---
```

Файл — **host bind-mount** (`./prompts.log:/app/prompts.log`,
[docker-compose.yml](../../../docker-compose.yml)), специально «чтобы читать не заходя в контейнер».
Лежит на хосте: `./prompts.log`. Rebuild образа его **не стирает**.

**Выбор целевой записи (важно — записей несколько):** в логе несколько `behaviorDecision`
вокруг эпизода (есть запись, где «Карл объяснись» = `[#1]`, и другая, где `#80`). Целевая —
та, чей `RESPONSE` совпадает с текстом сообщения `200` (ground truth из БД). Скрипт:

- ищет запись `behaviorDecision`/`behaviorDecisionEscalated`, чей `RESPONSE` содержит известный
  текст ответа `200` (передаём фрагмент-якорь аргументом/константой);
- **валидирует**: последний `[#N]`-блок behavior-messages содержит `[TRIGGER]` и текст
  «Карл объяснись». Если нет — падает с ошибкой (не угадываем).

Тип записи (`behaviorDecision` vs `behaviorDecisionEscalated`) сообщает, на какой модели был
исходный вызов; реплеим на той же.

### Структура behavior-messages в prompt

Все сообщения окна рендерятся в **один system-блок** как строки
`[#N] [userId:..] [username:..] [role:..] [source:..] <маркеры>\n<reply/quote>\n<content>`.
Маркеры: `[TRIGGER]`, `[GATE_CONTEXT]`, `[BATCH]`, адресация `[to:you]`/`[to:@user]`/`[to:room]`.
Скрипт находит этот блок как system-message, чей `content` содержит `[#`-строки / `[TRIGGER]`.

### Варианты (один на причину)

Мутации строятся **относительно строки `[TRIGGER]`** (без хардкода ординалов) — это зеркалит
предлагаемый фикс «последние N до триггера».

| Вар    | Причина            | Мутация                                                                          |
|--------|--------------------|----------------------------------------------------------------------------------|
| `V0`   | baseline           | без изменений (точный лог-prompt)                                                 |
| `V1`   | #1 topic-window    | навесить `[GATE_CONTEXT]` на header'ы последних N сообщений перед `[TRIGGER]`     |
| `V2`   | #2 время/пауза     | вставить видимую границу перед этими N: `--- новая пачка, пауза ~40 мин ---`      |
| `V3`   | #4 persistent state| убрать system-блоки personality/political/state (по сигнатуре их шаблонов)        |
| `V4`   | #5 адресация       | заменить `[to:room]`→`[to:you]` на строке `[TRIGGER]`                             |
| `Vall` | потолок            | `V1`+`V2`+`V4` вместе (дешёвые структурные правки)                                |

`N` — число сообщений свежей пачки до триггера (ориентир 5-10; параметр скрипта).

**Причина #3** (короткое анафоричное «объяснись») — не чистый ablation: менять текст юзера =
менять смысл. Оставляем как контроль, не как вариант.

### Прогон

- Каждый вариант — **K=10** прогонов.
- Модель — та же, что в целевой записи (`gpt-5.4-mini` default; `gpt-5.5` если запись была
  `behaviorDecisionEscalated`).
- **Структурированный вывод**: реальная `behaviorDecisionJsonSchema` через
  `client.chat.completions.parse` — точно как
  [`OpenAiSdkGateway.parseChatCompletion`](../../../src/infrastructure/external/OpenAiSdkGateway.ts).
- Клиент: `new OpenAI({ apiKey: process.env.OPENAI_KEY })` — без DI/бутстрапа бота.
- Из каждого выхода достаём текст `reply` (поле action-а решения).

### Судейство

**Ручное** (по решению пользователя). Скрипт **не** авто-классифицирует. Он дампит:

- для справки — исходный «плохой» ответ (`200`) и краткое описание свежей темы;
- по каждому варианту — K пронумерованных `reply`-текстов, сгруппированных и подписанных.

Пользователь читает и сам размечает каждый ответ как `fresh` / `ai_drift` / `other`, считает
drift-rate по вариантам и определяет доминанту.

## Исполнение (в контейнере, полный rebuild)

1. Добавить временный entry `ablation: './src/ablation.ts'` в
   [`rsbuild.config.ts`](../../../rsbuild.config.ts) рядом с `manual-job`.
2. Написать `src/ablation.ts` (логика выше).
3. `docker compose build app` — пересобрать образ `karl-bot:latest` (включает `dist/ablation.js`).
4. `docker compose run --rm app node dist/ablation.js /app/prompts.log > ablation-results.txt`
   — разовый контейнер, не трогает рабочего бота; `/app/prompts.log` виден через существующий
   bind-mount; `OPENAI_KEY` из `.env` (env_file). stdout → хостовый `ablation-results.txt`.
5. Прочитать `ablation-results.txt`, разметить вручную.
6. **Откат**: вернуть `rsbuild.config.ts`, удалить `src/ablation.ts`, удалить `ablation-results.txt`.
   Ничего не коммитим.

`prompts.log` бэкапить не нужно — он host bind-mount, rebuild его не трогает.

## Артефакт

- `ablation-results.txt` — дамп на хосте, local-only, не коммитим.
- `src/ablation.ts` + entry в `rsbuild.config.ts` — временные, откатываются после прогона.

## Риски / ограничения

- **Baseline не воспроизводит дрифт** → см. gate воспроизводимости; эксперимент инконклюзивен.
- **One-factor-at-a-time** не ловит взаимодействия причин → на это `Vall`.
- **`V2` без реальных timestamps** тестирует «сигнал границы/паузы», а не точные времена; это и
  есть гипотеза #2, но помним про приближение. (`messages` исходно без timestamp-колонки,
  миграция 001; при желании точные времена реконструируются из `behavior_events.created_at`.)
- **`gpt-5.4-mini` — reasoning-модель**, может игнорить часть параметров; зеркалим фактический
  вызов gateway (`parse`, та же схема), без отсебятины.
- **Выбор не той записи** → жёсткая валидация (RESPONSE-якорь + `[TRIGGER]`/«объяснись» в
  последнем блоке), падаем при несовпадении.

## Вне scope

- Сам **фикс** доминирующей причины — отдельный брейншторм после того, как доминанта известна.
